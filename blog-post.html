TL;DR — Problems & Fixes

Here's a quick rundown of the problems I ran into while building this:

  • stdout silently breaks MCP — Claude Code's stdio transport reserves stdout entirely for JSON-RPC. A single stray print() anywhere in the server corrupts the protocol with no error message. Fix: log everything to stderr only; logging.getLogger() writes to stderr by default.
  • Hooks blocking Claude's main loop — synchronous hooks add latency to every keypress. Fix: all hook scripts end with & to fire-and-forget into the background, posting to the daemon asynchronously.
  • Daemon not alive when hooks first fire — launchd loads the plist and immediately triggers the session-start hook before the virtualenv exists. Fix: added a venv existence check and retry loop in install.sh before registering any hooks.
  • Gemma's context window defaulting to 2K — Ollama's out-of-the-box context is 2048 tokens regardless of what the model supports. Fix: create a custom Modelfile with PARAMETER num_ctx 131072 and run ollama create gemma4:e4b -f Modelfile.
  • State files growing past the context window — long sessions push codebase.md large enough to overflow on injection. Fix: added a 100KB rotation limit in state_manager.py; old content rolls to an archive file.
  • Session-start injecting deleted files — codebase.md would list files that no longer existed, confusing Claude about the project structure. Fix: existence-prune the codebase map before injection so only real paths appear.

The Problem

Like most people, I've been using LLM's to write the majority of my code for a couple of years, specifically Codex and Claude. A couple of months ago, I noticed that Claude's usage limit would be reached significantly faster on my Pro plan than before while asking for very basic prompts, and it was confirmed by Anthropic that they were throttling limits. This became extremely annoying and frustrating when Claude would reach its limit after 1 prompt, and I wanted to find a solution.

After the Claude Code harness leak, I delved into how they compacted prompts and long chats. I'll write a more extensive blog on this later, but it uses an older version of Sonnet and a sliding-window algorithm to only retain and re-summarize certain parts of the conversation. However, this is extremely expensive, and one of the only ways of fixing this was simply starting a new chat and giving a handoff prompt, but this is time consuming and pretty expensive. I've solved that and added some features to make this free, efficient, and effective.

The Solution

This project uses an MCP server that plugs into Claude Code and handles compaction locally. Instead of asking Claude to summarize your session, it sends that work over HTTP to Ollama — a runtime for running LLMs on your own machine — where Gemma 4 E4B (a 4-billion parameter model from Google) does the summarization for free.

The architecture is simple:

Claude Code → stdio → MCP server → HTTP → Ollama (localhost:11434) → Gemma 4 E4B

Beyond compaction, the server also maintains lightweight state files (progress.md, codebase.md) that persist across sessions. At the start of each new Claude Code session, a compressed snapshot of your project state gets injected into context automatically, so Claude always knows where things left off without you having to re-explain everything.

The Tech

  • MCP Server: Python, using the fastmcp library. Exposes 6 tools to Claude Code over stdio.
  • Local LLM: Ollama running Gemma 4 E4B — 128K context window, runs entirely on Apple Silicon with no internet required.
  • Hooks: Claude Code lifecycle hooks (shell scripts) that fire on session start, prompt submit, file edit, and session stop to keep state files updated automatically.
  • Daemon: A lightweight background server (hook_runner.py --serve) on localhost:7737, started by launchd, that handles async hook calls without blocking Claude's main loop.
  • State: Plain markdown files in state/. No database, no external services, nothing that can go down.

The 6 tools the MCP server exposes are: compact_prompt, summarize_history, generate_handoff, log_event, get_context, and get_info. Most of the interesting work happens in the first three.

The diff Claude Code reserves stdout entirely for MCP JSON-RPC, so a single stray print() anywhere in the server silently corrupts the protocol. Hook calls also have to be non-blocking, else they would be calling Claude continuously and hence burning up tokens. The fix was using daemon to create a fire-and-forget POST to, which handles the actual state writes in the background. Here's what one hook call looks like: # on-edit.sh — fires after every file write, logs the change asynchronously INPUT=$(cat) curl -s -X POST http://localhost:7737/log \ -H "Content-Type: application/json" \ -d "{\"event_type\": \"edit\", \"content\": $INPUT}" & -d "{\"event_type\": \"edit\", \"content\": $INPUT}" &

Does It Work?

Yes, and considerably well — though the numbers are more honest than I expected. I ran a benchmark sending four representative session content types through Gemma 4 E4B and measuring token count before and after. The results were real, not estimated:

Whats next So far it works locally, as the MCP is only set up for the specific project. I'm working to make it a global MCP server so I can use it across projects and for all Claude instances automatically. Apart from using better models as they come along (smarter, faster, and easy to run), I want to dynamically budget the amount of information I can inject based on how complex the current task looks. There's also room to improve the codebase map, which right now is a per-file one-liner but could be graph-aware enough to surface which modules depend on each other. One thing I haven't solved completely is measuring whether Gemma's summaries actually preserve the right information. A “bad” compaction where it omits important information, or summarizes incorrectly can mislead Claude for the rest of the session, and I'd have no way to know. That's probably the most important open problem. github.com/sparshkapoor/promptcompactor.
